Dynamics Explorer covers "most users, simple question" but structurally cannot do two things still done in Akoya Go (poor UX): Track A = maintenance staff find a record and edit a field; Track B = volume users pull filtered data beyond the Explorer's agentic cap (trigger: CSO ask for ~5000 requests as Excel). **Two separate apps, shared spine** (AI proposes → deterministic code performs → human-confirm gate; compiler-not-interpreter, the opposite of Explorer). Full converged design + Codex reconciliation + `[VERIFIED file:line]` claims: **`docs/DATAVERSE_POWER_TOOLS_DESIGN.md`** — read it before any build plan.

**Why:** Track B's naïve "filter→page→Excel" framing is wrong. `akoya_request` is **polymorphic** ("grant" is a *view*: research grants, feedback-only concepts, SoCal program, high-volume discretionary, $100M projects) and **temporally polymorphic** (clean Akoya-native vs. approximate migrated; "ever" is rhetorical, deep-history is out of v1 by design → routes to Track A/Explorer find-one). Completeness = property of (field × era). Users are PhDs/lawyers/MBAs who own their analyses — optimize against the *plausible* wrong answer that passes the sniff test, not naïve trust; provenance = reproducible methods, not defensive armor. Fits [[project_intake_portal_skinny_scope]]; on the GOapply-replacement arc.

**How to apply:**
- Track A: populated-fields-only inline edit + single Dataverse Search box. v1 single-record, safe scalars only, optionset/lookup read-only. **Write path gated on 3 Codex-surfaced decisions**: (1) `MSCRMCallerID` attribution NOT guaranteed (env-gated, 403 retries without it, falls back to service principal) — fail-closed/warn; (2) `updateRecord` skips `checkRestriction` — enforce explicitly; (3) `IsValidForUpdate=true` insufficient — exclude calculated/rollup/formula via SourceType/AttributeOf. Read/display path unblocked.
- Track B: NOT a generic builder, NOT fixed templates — a **plain-English structured filter builder** (fan-out choices forced, e.g. "Budget: awarded/requested/total") + **mandatory composition/era/classification disclosure even in expert mode** + generous trust-bounded export (incl. discriminator/era/resolved-institution cols + explicit `UNKNOWN — not captured` sentinel) → refine in Excel. AI = phase-2 on-ramp for non-experts, not the core. Backoff-hardened paging is v1 (Codex). Parameterize the arbitrary 5000 cap; surface `capped`/`totalCount` in UI (helper already returns them).
- Real Track B work = 3 evidence-derived semantic-layer artifacts (plain-English field dict w/ fan-out; type/era taxonomy + boundary; field-availability boundary for ~10-15 export cols). **Design converged but BLOCKED on 3 evidence tasks before build plan:** Claude→Dataverse probe (discriminator field/values/volumes/era split via `docs/atlas/dataverse-akoya-request.md`); user→AkoyaGo excavation (trusted-view operational filters + migration cutover date); Connor→recent-era taxonomy + remap history.
- Discriminator/era probe DONE (S156, `scripts/probe-akoya-request-discriminators.js`): `akoya_request` true count **~25,561** but OData `/$count` caps at **5,000** (silent ~80% undercount — this IS the trigger; export engine MUST use FetchXML aggregate/RetrieveTotalRecordCount, never `$count`). Discriminator is a **composite**: `wmkf_request_type` (Request 16,227 / Concept 3,273 / Office-Site-Visit+Phone ~5,268 non-funding logs) × `wmkf_grantprogram` (Research 8,500 / Discretionary 5,345 / **null 4,634** / Southern California 4,489 / …) × `akoya_requesttype` (Grant 25,473 / Scholarship 88). All user-described slices confirmed w/ volume. Era: `createdon` 2023-dominated (22,573) = bulk migration import; NOT a native-era signal; cutover ~2023 (confirm exact w/ Connor). `overriddencreatedon` inconclusive. Full distributions folded into `docs/atlas/dataverse-akoya-request.md` + design doc Track B "Evidence".
- Audit/change-history (S156 probe `scripts/probe-dataverse-audit-capability.js`, Codex-reviewed): org+field auditing is ON, but the app SP lacks `ReadAuditSummary` so the *aggregate* "which fields edited most" query is unavailable; per-record `RetrieveRecordChangeHistory` returned 200 on ONE akoya_request (demonstrated, NOT generalized). Change-frequency is a **weak, biased proxy** (conflates Power Automate/workflow/rollup writes with human edits; era-sampling bias; privilege-fragile). **NOT a v1 input** — possible later inline point-of-edit enhancement only, gated on stratified verification (doc step 5). Do not build v1 field-prioritization on it.
