Dynamics Explorer covers "most users, simple question" but structurally cannot do two things still done in Akoya Go (poor UX): Track A = maintenance staff find a record and edit a field; Track B = volume users pull filtered data beyond the Explorer's agentic cap (trigger: CSO ask for ~5000 requests as Excel). **Two separate apps, shared spine** (AI proposes → deterministic code performs → human-confirm gate; compiler-not-interpreter, the opposite of Explorer). Full converged design + Codex reconciliation + `[VERIFIED file:line]` claims: **`docs/DATAVERSE_POWER_TOOLS_DESIGN.md`** — read it before any build plan.

**Why:** Track B's naïve "filter→page→Excel" framing is wrong. `akoya_request` is **polymorphic** ("grant" is a *view*: research grants, feedback-only concepts, SoCal program, high-volume discretionary, $100M projects) and **temporally polymorphic** (clean Akoya-native vs. approximate migrated; "ever" is rhetorical, deep-history is out of v1 by design → routes to Track A/Explorer find-one). Completeness = property of (field × era). Users are PhDs/lawyers/MBAs who own their analyses — optimize against the *plausible* wrong answer that passes the sniff test, not naïve trust; provenance = reproducible methods, not defensive armor. Fits [[project_intake_portal_skinny_scope]]; on the GOapply-replacement arc.
