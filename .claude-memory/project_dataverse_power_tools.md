Two separate apps, one spine. Dynamics Explorer covers "most users, simple question" but structurally cannot do two things still done in Akoya Go (poor UX): **Track A** = maintenance staff find a record and edit a field; **Track B** = volume users pull filtered data beyond the Explorer's agentic cap (trigger: CSO ask for ~5,000 requests as Excel). Shared spine = AI proposes → deterministic code performs → human-confirm gate (compiler-not-interpreter, the opposite of Explorer). **Full design, all evidence, and `[VERIFIED file:line]` claims live in `docs/DATAVERSE_POWER_TOOLS_DESIGN.md` — read it before any build plan; this entry is orientation only.**

**Why:** Track B's naïve "filter→page→Excel" framing is wrong. `akoya_request` (~25,561 rows) is **polymorphic** ("grant" is a view: research grants, feedback-only concepts, SoCal, discretionary, $100M projects) and **temporally polymorphic** (clean Akoya-native vs. approximate migrated). Completeness = property of (field × era × program). Users are PhDs/lawyers/MBAs who own their analyses — optimize against the *plausible* wrong answer that passes the sniff test. Fits [[project_intake_portal_skinny_scope]]; on the GOapply-replacement arc.

**How to apply:**
- Track A: populated-fields-only inline edit + single Dataverse Search box. v1 single-record, safe scalars only, optionset/lookup read-only. **Write path gated on 3 Codex-surfaced decisions:** (1) `MSCRMCallerID` attribution NOT guaranteed (env-gated, falls back to service principal) — fail-closed/warn; (2) `updateRecord` skips `checkRestriction` — enforce explicitly; (3) `IsValidForUpdate=true` insufficient — exclude calculated/rollup/formula via SourceType/AttributeOf. Read/display path unblocked.
- Track B: NOT a generic builder, NOT fixed templates — a plain-English structured filter builder (fan-out choices forced) + mandatory composition/era/classification/program disclosure even in expert mode + trust-bounded export with class-aware sentinels (never bare blanks). AI = phase-2 on-ramp for non-experts, not the core. **Export engine MUST use FetchXML aggregate / RetrieveTotalRecordCount, never OData `$count`** (caps silently at 5,000 — this undercount IS the trigger). Parameterize the arbitrary 5,000 cap; surface `capped`/`totalCount`.
- **Era classifier (exact, external-dependency-free, S157 dated evidence):** `createdon`=2023-12-03 ⇒ migrated (22,573, Blackbaud/"Sky"→AkoyaGO cutover, single ~43-min import window); `createdon`>2023-12-03 ⇒ Akoya-native (2,988). 🔴 **Exact as a CREATION-PROVENANCE partition only — NOT a business-era one** (`scripts/probe-akoya-codex-substantiation.js` C5): 168 native rows (18% of dated native) have `akoya_decisiondate`<2024, 169 (5.7% of native) some pre-2024 business date. Mandatory rule: time-slice history on `akoya_decisiondate` (corroborated by `wmkf_meetingdate`), NEVER `createdon`; treat migrated/native as provenance, not period; disclose the ~169-row cross-contamination when both dims appear. Creation-provenance findings (backfill/net-new/BB-lineage) are unaffected.
- **Decided-state predicate = `akoya_requeststatus` value→class map** (Pending\*=in-flight; \*Declined/Ineligible/Denied/Closed/\*Done/Approved=decided-terminal). NOT decisiondate-presence (that is an *approval* stamp — substantiated, not assumed). Ambiguous middle (`Active`/`Withdrawn`/`Proposal Not Invited`) behaviorally classified; only naming sign-off left for Connor.
