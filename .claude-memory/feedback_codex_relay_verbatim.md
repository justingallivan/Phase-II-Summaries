When relaying a Codex response (`codex:codex-rescue` subagent / `/codex:rescue` skill output) to the user, the user-visible message that delivers it must be **Codex's stdout verbatim** — no paraphrase, summary, re-ranking, or framing/analysis before or after it. The verbatim block is the *entire* delivery message. This holds for **every Codex round-trip in a session**, not just the first, and regardless of how the subagent was invoked (inline command path or direct `Agent(subagent_type: codex:codex-rescue)`). It is an explicit operating rule of the codex:rescue command (re-injected each run) and the `codex:codex-result-handling` skill contract; treat it as binding.

**Why:** Codex's exact wording, severity labels, and line numbers are what the user is paying tokens for — not my interpretation. Wrapper commentary editorializes the verdict before the user forms their own read of an independent review (e.g. recasting Codex's "document it" as "I have a better fix" in the same message steers interpretation and defeats the point of an independent pass). This applies even when the output seems verbose, repetitive, or summarizable — those judgments are not mine to make on Codex output; long is the right answer when the user wants verbatim.

**How to apply:** paste stdout exactly as returned inside a clearly-marked block, as the whole response. Acting on the findings afterward (fixes, commits, verification) is expected — that's doing the work, not commentary on the output. If a scope/decision question arises, raise it in a *separate subsequent turn*, never bolted onto the verbatim delivery.
