## 2. Verified Architectural Strengths

The following patterns represent high-integrity security boundaries within the application:

*   **Fail-Closed Identity Boundary:** The authentication flow in `pages/api/auth/[...nextauth].js` now explicitly returns `false` on any database errors during sign-in. This prevents the creation of "broken" sessions that lack verified profile metadata.
*   **Recursive OData Validation:** In `lib/services/dynamics-service.js` (Lines 151–185), the system employs a custom depth-tracking parser to validate `$expand` navigation properties. This prevents "Shadow Queries" where a user might attempt to sideload restricted data through a relationship.
*   **Centralized SSRF Protection:** The `lib/utils/safe-fetch.js` utility enforces a strict hostname allowlist and manual redirect validation for all outbound requests, mitigating Server-Side Request Forgery risks.
*   **Nonce-Based CSP:** The `middleware.js` dynamically generates a unique nonce per request, enforcing a strict Content Security Policy (CSP) that blocks unauthorized inline scripts and "unsafe-eval" in production.

## 4. Strategic Observations & Trade-offs

The audit identified three areas where security posture is balanced against operational requirements or policy-level decisions:

### A. Non-Browser CSRF (Intentional Design)
The CSRF validator in `lib/utils/auth.js` allows requests missing `Origin`/`Referer` headers to support Vercel cron jobs and server-to-server calls. While this is an intentional bypass for trusted non-browser clients, the team is prepared to implement a `CRON_SECRET` or "Header Allowlist" if stricter posture is required.

### B. AI Data Sovereignty (Policy Decision)
The Reviewer Finder transmits proposal text to Anthropic’s Claude API for analysis. This is a functional requirement. The security of this boundary relies on the **Anthropic Commercial Terms** (no training on customer data). The development team is exploring a "PII Scrubbing" layer as a future enhancement for highly sensitive proposals.

### C. Azure-Layer Enforcement (Pending Action)
The application currently enforces table/field restrictions at the code level. For maximal safety, it is recommended that IT configure the **Azure Service Principal** with a scoped Dataverse Security Role that is restricted to only the 19 required tables. This provides a hardware-level "fail-safe" even if application code is compromised.
