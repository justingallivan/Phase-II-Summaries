## Part 2: New Findings

1. [HIGH] Progress markers are not crash-safe after successful writes — `docs/BUDGET_FORM_SPEC.md:411`, `docs/BUDGET_FORM_SPEC.md:412` — Persist an idempotency key or alternate key per intended child row and/or marker-before-write state so a crash after a successful `createRecord` or `deleteRecord` but before marker persistence cannot duplicate or skip work on retry.

2. [HIGH] The accepted atomicity gap can last far longer than "seconds" under transient failure — `docs/BUDGET_FORM_SPEC.md:418`, `docs/BUDGET_FORM_SPEC.md:424`, `docs/INTAKE_PORTAL_DESIGN.md:323` — Reword the pilot acceptance to cover retry/backoff windows after children are written but before aggregate PATCH succeeds, or require `$batch` before production externalization.

3. [MODERATE] Permanent 4xx after partial delete/insert leaves no specified repair contract — `docs/BUDGET_FORM_SPEC.md:419`, `docs/INTAKE_PORTAL_DESIGN.md:345` — Define the force re-drain operation precisely, including whether it re-queries Dataverse, deletes inserted replacement rows, resets `dynamics_patches`, and replays from the frozen payload.

4. [MODERATE] Drain-time $100K permanent failure strands applicants on client-validation bugs — `docs/BUDGET_FORM_SPEC.md:409`, `docs/BUDGET_FORM_SPEC.md:420`, `docs/INTAKE_PORTAL_DESIGN.md:332` — Add a staff/applicant recovery path that reopens the draft or creates a corrected replacement job from the frozen payload instead of only marking the job failed and notifying staff.

5. [MODERATE] Audit target entity is wrong for several budget events — `docs/BUDGET_FORM_SPEC.md:293`, `docs/BUDGET_FORM_SPEC.md:299`, `lib/services/intake-audit-service.js:90` — Set `targetEntity`/`targetId` per event (`intake_drafts` for autosave, `submission_jobs` for enqueue/drain failure, `akoya_request` for validation and aggregate externalization) so `queryByTarget()` returns meaningful audit history.

6. [MODERATE] App-user impersonation claim omits the live feature flag and fallback — `docs/BUDGET_FORM_SPEC.md:307`, `docs/BUDGET_FORM_SPEC.md:412`, `lib/services/dynamics-service.js:166`, `lib/services/dynamics-service.js:191` — State that `DYNAMICS_IMPERSONATION_ENABLED` must be true and decide whether budget drains accept the existing 403 fallback to service-principal attribution or need a no-fallback write path.

7. [LOW] App-user `systemuserid` source is not anchored in a versioned repo document — `docs/BUDGET_FORM_SPEC.md:307`, `docs/BUDGET_FORM_SPEC.md:516`, `docs/DYNAMICS_IDENTITY_RECONCILIATION_PLAN.md:78` — Replace the memory-only reference with a concrete Atlas field, environment variable, or resolver query that the drain implementation can read before its first write.

8. [LOW] New proposal-budget fields still need the catalog process made explicit — `docs/BUDGET_FORM_SPEC.md:341`, `docs/BUDGET_FORM_SPEC.md:510`, `docs/INTAKE_PORTAL_SCHEMA_CHANGES.md:16` — Update `docs/INTAKE_PORTAL_SCHEMA_CHANGES.md` during the schema slice to add `wmkf_rolecode`, `wmkf_headcount`, and `wmkf_effortpct` to the queued catalog before generating schema JSON.

9. [NIT] Category collapse is acceptable only because role code and description preserve row identity — `docs/BUDGET_FORM_SPEC.md:324`, `docs/BUDGET_FORM_SPEC.md:335`, `docs/BUDGET_FORM_SPEC.md:353` — State explicitly that Connor's PA must use `wmkf_description` or `wmkf_rolecode` for detailed reviewer lines, not `wmkf_category` alone, because animal costs, contracted services, renovations, and dynamic other operations all collapse to `Other Direct`.

## Part 3: Summary

v2 is materially stronger than v1: the externalized shape now matches the pre-committed catalog, audit coverage exists, the drain lock boundary is explicit, and the spec no longer pretends `$batch` exists. The top remaining risks are concentrated in the non-batch drain: progress markers are not fully crash-safe, the "seconds-wide" aggregate gap can stretch across retry backoff, and permanent-failure recovery is still procedural rather than specified. Recommended next action: tighten the `dynamics_patched` idempotency/repair contract and attribution requirements before schema JSON or drain implementation begins.
